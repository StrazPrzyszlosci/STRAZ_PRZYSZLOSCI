# Plan: Discord Bot o zbliżonej funkcjonalności do Telegram Bot

## Cel

Na bazie istniejącego bota Telegram (`cloudflare/src/telegram_issues.js` + `telegram_ai.js`), uruchomić bota Discord o porównywalnej funkcjonalności: AI chat, GitHub Issues, skanowanie części/datasheetów, odczyt rezystorów, katalog recyklingowy.

## Obecna architektura (Telegram)

```
Telegram API (webhook) 
  → Cloudflare Worker (worker.js)
    → telegram_issues.js:  routing + webhook handling + callback queries
    → telegram_ai.js:       AI logic (Gemini/NVIDIA), moderation, sessions, scanning, datasheets
    → telegram_utils.js:    sendMessage, Markdown sanitization, main menu keyboard
    → D1 Database:          chat messages, user sessions, throttling, recycled parts
```

## Decyzja architektoniczna — Discord Gateway vs HTTP Interactions

Discord oferuje dwa modele komunikacji:
1. **Gateway API** (WebSocket) — pełen dostęp do wiadomości, eventów, komend tekstowych. Wymaga długotrwałego połączenia.
2. **HTTP Interactions** — tylko slash commands, przychodzące jako HTTP POST (działa z Cloudflare Worker'em, ale **ograniczone do komend `/`, nie czyta zwykłych wiadomości**).

**Decyzja**: Discord bot z Gateway API. Tylko to daje funkcjonalność porównywalną z Telegram botem (AI chat na dowolną wiadomość, obsługa zdjęć, sesje interaktywne). HTTP Interactions są zbyt ograniczone dla tego use case.

To oznacza, że Discord bot musi działać jako **osobny proces** poza Cloudflare Worker'em — nie można utrzymywać połączenia WebSocket wewnątrz Worker'a (limit 30s CPU).

### Opcje hostingu dla Discord bota

| Opcja | Zalety | Wady |
|-------|--------|------|
| Docker container na VPS | Pełna kontrola, trwałe połączenie | Więcej infra do zarządzania |
| Cloudflare Workers z Durable Objects (WebSocket) | W ekosystemie CF | Durable Objects to nowy paradygmat, złożoność |
| Ten sam Worker jako proxy AI | Prosty | Gateway i tak potrzebuje osobnego hosta |

**Rekomendacja**: Nowy plik `cloudflare/src/discord_bot.mjs` jako **samodzielny proces Node.js** (np. uruchamiany jako drugi Cloudflare Worker script przez cron/D1 trigger, albo Docker). Używa discord.js v14 do Gateway. Dla wywołań AI może używać własnych kluczy API (Gemini/NVIDIA) lub wołać istniejącego Worker'a jako proxy.

## Zakres implementacji

### 1. Nowy plik: `cloudflare/src/discord_bot.mjs`

Samodzielny proces używający discord.js v14, który:

- Łączy się z Discord Gateway (Intents: `GuildMessages`, `MessageContent`, `Guilds`)
- Reaguje na wiadomości w kanałach (z allowlistą `DISCORD_ALLOWED_CHANNEL_IDS` lub `*`)
- Reaguje na komendy tekstowe (`/start`, `/help`, `/reset`, `/stop`, `/menu`)
- Reaguje na interakcje z przyciskami (Discord Buttons, Select Menus)
- Obsługuje załączniki (zdjęcia, PDF — przez Discord CDN, odpowiednik `fetchTelegramFileAsBase64`)

**Mapowanie Telegram → Discord**:
| Telegram | Discord |
|----------|---------|
| `chat_id` | `channel_id` |
| `message.message_id` | `message.id` |
| `message.from.id` | `message.author.id` |
| `message.text` | `message.content` |
| `message.file_id` + `fetchTelegramFileAsBase64` | `message.attachments` → download z URL |
| `sendTelegramReply` + `reply_to_message_id` | `message.reply()` |
| Telegram Inline Keyboard (callback_data) | Discord ActionRow + Button (customId) |
| `/command` | prefix `!komenda` lub slash command |

### 2. Menu Discord vs Telegram

Telegram używa `inline_keyboard` (przyciski pod każdą wiadomością). Discord używa:
- **Message Components** (Buttons, Select Menus) — odpowiedniki inline keyboard
- **Slash Commands** — uzupełnienie

Discord bot będzie obsługiwał oba mechanizmy, ale dla zachowania podobnego UX główny interfejs to Message Components.

### 3. Współdzielenie logiki

Zamiast kopiować kod, **ekstrahujemy wspólne funkcje** z `telegram_ai.js` do oddzielnego modułu. Wspólne elementy:

- `generateChatReply(apiKeys, message, history)` — generowanie odpowiedzi AI (przyjmuje klucze API zamiast `env`)
- `moderateIssueCandidate(apiKeys, classification, message, history)`
- `draftIssueBody(apiKeys, classification, history)`
- `recognizeDeviceAndListParts(apiKeys, base64)` — computer vision
- `recognizePartForScanFlow(apiKeys, base64, options)` — rozpoznawanie części
- `handleResistorAnalysis(apiKeys, message)` — odczyt rezystorów
- `routeTelegramIntent` → `classifyMessageIntent(text)` — klasyfikacja intencji (bez zależności Telegram)

Funkcje sesji (już wyeksportowane z `telegram_ai.js`):
- `getUserSession`, `upsertUserSession`, `closeUserSession`, `closeAllUserSessions`
- `loadTelegramChatHistory` → `loadChatHistory`
- `checkTelegramChatRateLimit` → `checkChatRateLimit`

Funkcje bazy danych (używają `env.DB` — D1):
- Dla Discord możemy użyć tej samej bazy D1 przez HTTP API Cloudflare, lub wydzielić warstwę abstrakcji.

**Kluczowa decyzja**: Czy Discord bot ma dostęp do tej samej bazy D1 co Telegram Worker?

- **TAK** (rekomendowane): Discord bot wywołuje operacje na D1 przez HTTP forwarding do Worker'a (dodajemy endpointy REST dla botów na `/v1/bot-sessions/*`, `/v1/bot-history/*` itp.) — albo bezpośrednio przez Cloudflare D1 HTTP API.
- **NIE** (prostsze, ale tracimy współdzielenie stanu): Discord bot ma własny storage (np. SQLite lokalnie). Sesje i historia są odseparowane, użytkownicy nie dzielą stanu między platformami.

### 6. Deployment

Discord bot będzie uruchamiany jako osobny proces, np.:

```bash
# Lokalnie
node cloudflare/src/discord_bot.mjs

# Na VPS
pm2 start cloudflare/src/discord_bot.mjs --name discord-straz

# Jako Docker
docker build -t discord-straz-bot .
docker run -d --env-file .env.discord discord-straz-bot
```

Alternatywnie, można użyć Cloudflare Workers z triggerem cron + KV do symulacji Gateway, ale to złożone i niezalecane.

## Potencjalne konflikty dwóch botów

### 1. Zanieczyszczenie historii czatu w D1
Tabela `telegram_chat_messages` jest kluczowana po `chat_id` (Telegram). Discord używałby `discord_channel_id` jako klucza. **Rozwiązanie**: dodać kolumnę `platform` (`telegram` / `discord`), poszerzyć klucz o platformę, i nie mieszać historii.

### 2. Kolizje sesji użytkownika
Tabela `user_sessions` kluczowana po `(chat_id, user_id, session_type)`. Dla Discord: `(channel_id, author_id, session_type)`. Dla Telegram: `(chat_id, telegram_user_id, session_type)`. Ponieważ ID chatów Telegram to liczby typu `123456789`, a Discord use `discord_channel_id` jako string Snowflake (`987654321098765432`), zakresy się nie pokrywają liczbowo, ale obie platformy mogą mieć ID liczbowe. **Rozwiązanie**: prefixować klucze platformą: `tg:123456789:987654` vs `dc:987654321098765432:123456789`.

### 3. AI Rate Limiting
Oba boty współdzielą te same klucze API Gemini/NVIDIA. Jeśli oba są aktywne jednocześnie, mogą szybciej wyczerpać limity (domyślnie: 10 req/5min + 60 req/dzień). **Rozwiązanie**: współdzielić te same tabele rate-limit (albo osobne z tymi samymi limitami globalnymi), albo zwiększyć limity w konsoli Google/NVIDIA.

### 4. GitHub Issues — podwójne zgłoszenia
Ta sama osoba może zgłosić ten sam pomysł z Telegrama i Discorda. Brak deduplikacji. **Rozwiązanie**: dodać label `discord` i `telegram` do Issues, żeby rozróżnić źródło. Dla moderacji: rozważyć prostą deduplikację po podobieństwie tytułu (opcjonalne, później).

### 5. Deployment complexity
Dwa procesy wymagają:
- Monitoringu (healthcheck, logi, restarty)
- Osobnego CI/CD
- Osobnej konfiguracji sekretów

**Rozwiązanie**: Discord bot może logować do tego samego systemu co Worker (Cloudflare Logpush lub oddzielny plik). Proces Node.js może być zarządzany przez PM2 lub Docker z restart-policy.

### 6. Token management
Więcej sekretów do zarządzania: `DISCORD_BOT_TOKEN`, `DISCORD_APPLICATION_ID`. **Rozwiązanie**: trzymać w `.env` lub w Cloudflare Secrets (jeśli Discord proxy używa Worker'a jako proxy).

### 7. Spójność formatu odpowiedzi
Telegram używa Markdown z ograniczonym formatowaniem. Discord używa własnego formatu (podobny, ale inny — `**bold` vs `*bold*`, brak `\`` dla inline code w tytule embeda). **Rozwiązanie**: Discord bot będzie formatował odpowiedzi w Discord Markdown, a funkcje AI generujące tekst będą dostawały parametr `platform` (np. `telegram` vs `discord`).

### 8. Długość wiadomości
Telegram: max 4096 znaków. Discord: max 2000 znaków na zwykłą wiadomość, 4096 w embedach. **Rozwiązanie**: Discord bot będzie automatycznie dzielił długie odpowiedzi na embedy lub wiele wiadomości.

## Plan implementacji (kroki)

### Krok 1: Ekstrakcja wspólnej logiki AI
- Utworzyć `cloudflare/src/shared_ai.js`
- Przenieść tam funkcje czysto-logiczne (bez zależności od `env.DB`, `env.AI`, `env.TELEGRAM_*`)
- Obie platformy będą importować z tego modułu

### Krok 2: Abstrakcja dostępu do bazy danych
- Utworzyć `cloudflare/src/db_adapter.js` z interfejsem `PlatformDB`
- Implementacja dla Telegram Worker (bezpośrednio `env.DB`)
- Implementacja dla Discord bota (przez HTTP API do Worker'a)

### Krok 3: Discord Bot — główny plik
- `cloudflare/src/discord_bot.mjs`:
  - Inicjalizacja discord.js Client z odpowiednimi Intentami
  - `messageCreate` handler → mapowanie na tę samą logikę co Telegram
  - `interactionCreate` handler → obsługa Button/Select Menu
  - Mapper `discordMessageToStandardMessage(discordMsg) → { chat_id, user_id, text, file_base64, ... }`
  - Ta sama pętla: intencja → processCommand / processIssue / processConversation

### Krok 4: Discord Button Handlers
- Odpowiedniki Telegram `CALLBACK_HANDLERS` dla Discord komponentów:
  - `menu_scan`, `scan_part_start`, `scan_part_edit`, `scan_part_add`
  - `scan_batch_start`, `scan_batch_enter_model`, itd.
  - `menu_datasheet`, `menu_search`, `menu_issue`, `menu_resistor`
  - `recycled_*` handlers
  - `cancel_session`, `command_start`

### Krok 5: Discord Main Menu
- Discord nie ma "klawiatury" pod każdą wiadomością jak Telegram
- Alternatywy:
  - **ActionRow z Buttonami** w każdej odpowiedzi (najbliższe Telegram)
  - Slash commands (dodatkowo)
  - Hybrid: główne menu jako stała wiadomość z buttonami (tzw. "dashboard message")

### Krok 6: Obsługa załączników (zdjęcia, PDF)
- Discord: `message.attachments` → pobranie URL → fetch → base64
- Odpowiednik `fetchTelegramFileAsBase64` → `fetchDiscordAttachmentAsBase64`

### Krok 7: Wdróżenie
- Stworzyć `package.json` zależności dla Discord modułu (discord.js v14)
- Stworzyć `Dockerfile` lub instrukcję PM2
- Testy smoke: podobne do `telegram_issue_smoke_test.py`

### Krok 8: Dokumentacja
- Nowy `docs/ARCHITEKTURA_MOSTU_DISCORD_GITHUB_ISSUES.md`
- Nowy `docs/RUNBOOK_URUCHOMIENIA_DISCORD_ISSUES.md`
- Aktualizacja `README.md`
